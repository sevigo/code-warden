# Indexing Pipeline

This document explains how Code-Warden indexes repositories into Qdrant — what gets stored, how it's structured, and how to debug retrieval issues.

---

## Overview

Code-Warden builds a searchable index of each repository in Qdrant. The index is used by the RAG pipeline at review time to retrieve relevant context — architecture summaries, symbol definitions, affected downstream code, and tests.

There are two indexing operations:

| Operation | Command | When to use |
|---|---|---|
| **Prescan** | `warden-cli prescan <repo-path>` | Initial full index, or forced rebuild |
| **Update** | `warden-cli update <repo-path>` | Incremental update after a git pull |

The `/review` command automatically runs an incremental update on changed files before generating a review. Prescan must be run manually for new repositories.

## Chunk Types

Every document stored in Qdrant has a `chunk_type` metadata field. This is the primary way the RAG pipeline targets specific retrieval stages.

### `code`

Regular code chunks — the main unit of indexing. Each chunk covers a portion of a source file, split at function boundaries where possible.

**Key metadata:**
```json
{
  "chunk_type": "code",
  "source": "internal/rag/service.go",
  "line": 132,
  "end_line": 158,
  "language": "go",
  "is_test": false,
  "package_name": "rag",
  "symbols": ["NewService", "Service"],
  "imports": ["context", "github.com/sevigo/goframe/..."]
}
```

### `definition`

Extracted type and function definitions — structs, interfaces, function signatures. These are stored separately from regular code chunks so they can be retrieved by exact symbol name without relying on semantic similarity.

**Key metadata:**
```json
{
  "chunk_type": "definition",
  "source": "internal/rag/service.go",
  "identifier": "Service",
  "kind": "interface",
  "package_name": "rag",
  "is_exported": true,
  "line": 34,
  "end_line": 47
}
```

The definitions stage in the RAG pipeline first tries an exact Qdrant filter on `identifier` before falling back to semantic search.

### `arch`

Pre-computed architectural summaries — one per directory. Generated by the prescan process using an LLM to summarize all files in a directory. These give the review model high-level module understanding without needing to retrieve individual files.

**Key metadata:**
```json
{
  "chunk_type": "arch",
  "source": "internal/rag/",
  "directory": "internal/rag"
}
```

### `toc`

Table-of-contents entries — a compact representation of each file listing its exported symbols and imports. Used by the description context stage to find relevant files quickly.

**Key metadata:**
```json
{
  "chunk_type": "toc",
  "source": "internal/rag/service.go",
  "package_name": "rag",
  "exports": ["NewService", "Service", "Config"],
  "imports": ["context", "fmt"]
}
```

### Test chunks (`is_test: true`)

Test files produce `code` chunks with `is_test: true`. They are filtered out of regular retrieval (impact, description stages) and only surfaced through the dedicated test coverage stage, which matches on `tested_symbols` metadata.

```json
{
  "chunk_type": "code",
  "is_test": true,
  "tested_symbols": ["UserService", "CreateUser"],
  "source_file": "internal/rag/service.go"
}
```

## Sparse Vectors

Every chunk is stored with both a dense embedding and a sparse vector. The sparse vector is generated by the code-aware tokenizer:

- Splits camelCase (`processPayment` → `process`, `payment`)
- Splits snake_case (`get_user` → `get`, `user`)
- Handles acronyms (`XMLParser` → `xml`, `parser`)
- Filters language keywords (`func`, `return`, `if`, etc.)
- Hashes tokens via FNV32a into a 50,000-dimension sparse space

At query time, the same tokenizer runs on the search query and both dense and sparse scores are combined (hybrid search). This improves recall for exact symbol names and identifiers that semantic similarity alone tends to miss.

The sparse vector field in Qdrant is named `code_sparse`. This must match the collection schema — if you change `ai.sparse_vector_name` in config, you need to re-index all collections.

## Incremental vs Full Indexing

**Incremental (`update`):**
- Computes `git diff` between the stored SHA and HEAD
- Re-indexes only changed files
- Removes old chunks for deleted/renamed files
- Fast — typically a few seconds for a PR-sized change

**Full (`prescan`):**
- Walks the entire repository file tree
- Skips files matching `exclude_dirs` / `exclude_exts` from `.code-warden.yml`
- Processes files in batches with configurable concurrency
- Resumable — tracks progress so a killed prescan can continue from where it stopped
- Generates `arch` summaries per directory at the end

After prescan completes, `update` is sufficient for keeping the index fresh.

## What gets skipped

Files are skipped during indexing if they match:

- `exclude_dirs` in `.code-warden.yml` (e.g. `vendor`, `node_modules`, `dist`)
- `exclude_exts` in `.code-warden.yml` (e.g. `.md`, `.txt`, `.lock`)
- Binary files (detected by content sniffing)
- Files exceeding the maximum file size limit

Configure per-repository exclusions in `.code-warden.yml`:

```yaml
exclude_dirs:
  - vendor
  - node_modules
  - testdata

exclude_exts:
  - .md
  - .lock
  - .sum
```

## Debugging Retrieval Issues

### "Why isn't my symbol being found?"

1. **Check the definition chunk exists:**
   Query Qdrant directly for the symbol:
   ```sh
   curl -X POST http://localhost:6333/collections/<repo-collection>/points/scroll \
     -H 'Content-Type: application/json' \
     -d '{"filter": {"must": [{"key": "chunk_type", "match": {"value": "definition"}}, {"key": "identifier", "match": {"value": "YourSymbol"}}]}, "limit": 5}'
   ```
   If no results, the symbol wasn't extracted during indexing. Check that the file is not excluded and that the language parser supports it.

2. **Re-index the file:**
   ```sh
   ./bin/warden-cli update /path/to/repo
   ```

3. **Check the collection exists:**
   ```sh
   curl http://localhost:6333/collections
   ```

### "Reviews are missing context for a package"

The `arch` chunk for that directory may be missing. Run a full prescan — arch summaries are only generated during prescan, not incremental updates.

### "Sparse vector generation failed" in logs

The code-aware sparse tokenizer is running but failing on specific input. Check the log for the token that caused the failure. This is non-fatal — the chunk falls back to dense-only search.

### "Context is empty" in reviews

The retrieval returned no documents. Possible causes:
- Repository not indexed yet — run prescan
- Collection name mismatch — check Qdrant collections match the configured repo
- Embedder model not running — verify Ollama is accessible and the model is loaded
- Query budget too low — check `ai.context_token_budget` in config

### Checking collection health

```sh
# List all collections and their sizes
curl http://localhost:6333/collections | jq '.result.collections[] | {name: .name}'

# Check a specific collection's point count
curl http://localhost:6333/collections/<collection-name> | jq '.result.points_count'
```

A healthy index for a medium-sized repo (50–200 Go files) typically contains 5,000–30,000 points.

## Qdrant Collection Schema

Each repository gets its own Qdrant collection named after the repository (`owner-repo` format). The collection is created automatically on first indexing with:

- A dense vector field using the configured embedding dimension
- A sparse vector field named `code_sparse` (configurable via `ai.sparse_vector_name`)
- Payload indexing on `chunk_type`, `identifier`, `source`, `is_test` for fast filtering

## Internal Code-Warden Indexing Strategy

To ensure the RAG pipeline can answer questions about its own internals, the indexing strategy follows these rules for system-critical components (e.g., `Indexer.ProcessFile`, `DefinitionExtractor`):

1. **Explicit Symbol Extraction**: Defining symbols like `Indexer` or `DefinitionExtractor` ensures they are stored as `chunk_type: "definition"` points.
2. **Multi-Stage Retrieval**: When a query contains system-level keywords (e.g., `scan`, `symbols`, `indexing`), the retrieval pipeline explicitly pulls from the `definition` chunk set in addition to directory-level `arch` summaries.
3. **Prescan Awareness**: Architectural summaries (`arch`) explain the *what* and *where*, while definition chunks explain the *how*. Both are required for the LLM to provide implementation-specific answers about Code-Warden's own codebase.
